## Authoring guidelines (read before writing a new hook)

- **Robust regex, never one fixed phrase (HARD).** A detection hook must match a CLASS of phrasings, not the single sentence that prompted it. Build the pattern from interchangeable PARTS combined combinatorially — e.g. `(commitment) × (action verb) × (deferral marker)` — so paraphrases still trip it. A brittle single-phrase regex gets dodged the moment the wording shifts. (2026-06-16, `ross-perot-guard.mjs` deferred-fix check.)
- **Account for the obvious variants:** straight AND curly apostrophes (`['’]`), hyphen/space/optional forms (`follow-?up`, `sign-?off`), case-insensitive (`/i`), and a bounded proximity window between parts (`[^.]{0,60}?`) instead of exact adjacency.
- **Gate to cut false positives:** only fire in the context that matters (e.g. builder-mode = a Write/Edit this turn), and give an explicit escape (a stated reason and/or an `<name>-override:` token).
- **Lock with a test (HARD).** Every hook ships a `*.test.mjs` sibling that includes **≥3 differently-worded positives** the hook MUST catch AND the false-positive cases it must NOT catch, plus each escape hatch. Red-before-green.
- **Fail open.** Wrap in try/catch and `exit(0)` on any error — a buggy hook must never block all work.
- **Register + document:** add it to `settings.json` (right event/matcher) and add a row below (the `hookbook-sync` Stop hook enforces both).
